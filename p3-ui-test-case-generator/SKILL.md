# UI 测试用例生成器

从功能地图 + 功能测试用例自动生成场景化 UI 自动化测试用例，每个 UI 场景映射到对应的功能用例 TC-ID，实现追溯。

**本 skill 负责生成用例，生成的 YAML 文件可直接用 p4-browser-test-runner-devtools 执行。**

## 前置条件

- 已有 `p1-web-explorer` 生成的功能地图 Markdown 文件（包含页面结构、筛选条件、数据列表等）
- 已有 `p2-test-case-generator` 生成的功能测试用例 Markdown 文件（包含 TC-ID、模块、优先级等）
- 业务系统 URL
- 登录账号密码（如需登录）

### 步骤 2：解析输入文件

#### 2.1 解析功能地图

读取功能地图 Markdown 文件，提取以下结构化信息：

| 提取维度 | 来源章节 | 用于 |
|----------|---------|------|
| 页面清单（名称、URL、类型） | 第一章 页面清单与URL | 场景入口地址 |
| 导航菜单（左侧菜单、面包屑） | 第二章 全局导航体系 | 导航步骤生成 |
| 筛选条件（字段、类型、选项值） | 第三章各页面的筛选条件 | 查询场景生成 |
| 操作按钮（名称、状态规则） | 第三章各页面的操作按钮 | 点击步骤生成 |
| 数据列表（列名、操作列） | 第三章各页面的数据列表 | 列表验证步骤 |
| 表单字段（名称、类型、必填、默认值） | 第三章新增/编辑页的表单字段 | 表单填写步骤 |
| Tab 页签 | 第三章配置页等 | Tab 切换步骤 |
| 页面跳转关系 | 第四章 页面跳转关系 | 场景依赖分析 |
| 业务流程 | 第五章 核心业务流程 | 端到端场景生成 |
| 状态流转 | 第六章 数据状态流转 | 状态验证步骤 |
| 枚举值汇总 | 第七章 关键枚举值汇总 | 测试数据准备 |

#### 2.2 解析功能测试用例

读取功能测试用例 Markdown 文件，提取：

| 提取维度 | 用于 |
|----------|------|
| TC-ID（如 TC-001、TC-002） | UI 场景追溯映射 |
| 功能模块（如 处罚任务提交、任务列表查询） | 按模块分组 |
| 优先级（P0/P1/P2） | 报告中的优先级展示 |
| 测试类型（功能/API/集成/性能） | 标记 UI 可自动化用例 |
| 用例描述 | 关联到 UI 场景步骤 |
| 前置条件 | 场景依赖分析 |
| 测试步骤 | 生成 UI 操作步骤的参考 |
| 预期结果 | 生成断言的参考 |

#### 2.3 建立 TC-ID 与 UI 场景的映射

将功能测试用例中的 TC-ID 映射到 UI 测试场景。映射规则：
- **1 个 UI 场景对应多个功能用例**：一个端到端 UI 场景可能覆盖多个功能用例（如"新增处罚"UI 场景覆盖 TC-001 正常提交 + TC-002 必填校验）
- **按模块归组**：同一模块的 TC-ID 尽量归入同一个或相邻的 UI 场景
- **按页面归组**：操作同一页面的功能用例归入同一场景
- **未覆盖标记**：功能地图中未找到对应页面的 TC-ID 标记为"需手工测试"

### 步骤 3：生成测试场景

基于功能地图信息，自动生成测试场景。

**场景类型与生成规则**：

#### A. 列表页场景（每个列表页生成）

对功能地图中类型为"列表页"的页面，生成以下场景：

| 场景模式 | 操作步骤 | 验证 |
|----------|---------|------|
| 打开页面 | 导航到目标页面 | 页面加载，列表展示数据 |
| 筛选查询 | 逐个填写筛选条件 → 点击查询 | 列表数据更新 |
| 重置筛选 | 点击重置按钮 | 筛选条件清空，列表恢复 |
| 下拉筛选 | 展开下拉框 → 选择每个选项 → 查询 | 每次查询结果变化 |
| 分页操作 | 点击下一页/上一页/指定页码 | 页面切换成功 |
| 切换每页条数 | 修改每页条数下拉 | 列表数量变化 |

#### B. 新增/编辑表单场景

对功能地图中类型为"表单页"的页面，生成以下场景：

| 场景模式 | 操作步骤 | 验证 |
|----------|---------|------|
| 正常提交 | 填写所有必填字段 → 提交 | 提交成功提示或页面跳转 |
| 必填校验 | 不填必填字段 → 提交 | 表单提示必填项 |
| 下拉选择 | 展开下拉框 → 逐个选择选项 | 选项正确展示 |
| 字符上限 | 输入框输入超长文本 | 字符计数或截断 |

#### C. 详情查看场景

依赖列表页场景创建的数据，生成：

| 场景模式 | 操作步骤 | 验证 |
|----------|---------|------|
| 进入详情 | 列表页点击查看按钮 | 详情页加载，信息展示完整 |
| 详情内列表 | 详情页内的子列表查询/筛选 | 子列表数据展示 |
| 操作按钮 | 点击各操作按钮（申诉/撤销等） | 弹窗或状态变化 |

#### D. 端到端业务流场景

根据功能地图第五章的"核心业务流程"，生成跨页面场景：

```text
示例：处罚任务完整流程
S1: 新增处罚 → S2: 列表查询验证 → S3: 查看任务详情 → S4: 处罚明细操作
```

#### E. Tab 切换场景

对含 Tab 页签的页面，生成 Tab 切换验证：

| 场景模式 | 操作步骤 | 验证 |
|----------|---------|------|
| Tab 切换 | 点击每个 Tab | Tab 内容切换 |
| Tab 内操作 | 在每个 Tab 内执行查询/操作 | 功能正常 |

#### F. 数据导出场景

对有"导出数据"按钮的页面，生成：

| 场景模式 | 操作步骤 | 验证 |
|----------|---------|------|
| 导出数据 | 点击导出按钮 | 文件下载成功 |

**场景编号规则**：S 前缀 + 编号（S1、S2...），按依赖关系排列，被依赖的场景排前面。

### 步骤 4：编排执行顺序

根据功能地图第四章"页面跳转关系"和第五章"业务流程"，分析场景依赖：

**依赖类型**：
- **数据依赖**：需要先创建数据才能查询/查看（S1 新增 → S2 查询 → S3 详情）
- **状态依赖**：需要先触发操作才能验证状态变化
- **无依赖**：可独立执行（如导出、分页）

**编排规则**：
1. 无依赖的场景优先
2. 有依赖的场景按拓扑排序
3. 每个场景标注 `depends_on` 和 `provides`（为下游提供什么数据）

输出执行顺序表：

```markdown
| 顺序 | 场景 | 依赖 | 页面入口 | 说明 |
|------|------|------|---------|------|
| 1 | S1: 处罚任务创建 | 无 | /penaltyTask/modify-event | 创建测试数据 |
| 2 | S2: 任务列表查询 | S1 | /penaltyTask | 查询S1创建的数据 |
| 3 | S3: 任务详情查看 | S1 | /penaltyTask/detail/{id} | 从S2进入详情 |
```

### 步骤 5：生成 YAML 测试步骤

为每个场景的具体步骤生成 browser-test-agent 格式的 YAML。

**步骤描述规范**：

功能地图提供了精确的元素信息，YAML 步骤描述直接使用：

| 功能地图信息 | YAML 步骤描述示例 |
|-------------|-----------------|
| 筛选字段"处罚主体"，下拉选项"商品" | `action: select, description: "处罚主体下拉框选择'商品'"` |
| 按钮"新增处罚" | `action: click, description: "点击新增处罚按钮"` |
| 表单字段"任务名称"，文本输入 | `action: input, description: "任务名称输入框", value: "自动化测试任务"` |
| 数据列"执行状态" | `action: assert, description: "列表中显示执行状态列"` |

**断言原则**：
- **操作写具体**：用功能地图中的实际元素名称（按钮名、字段名、选项名）
- **断言写宽泛**：只验证页面状态变化，不写死具体文案和数量
- **步骤编号**：`S{场景号}-{序号}`，如 S1-01、S1-02

### 步骤 6：输出文件

#### 6.1 输出目录结构

```
<输出目录>/
├── {项目名}-UI测试用例.md          # 场景化测试用例文档（含 TC-ID 追溯矩阵）
└── tests/
    ├── suite.yaml                  # 测试套件（含登录、执行顺序、依赖）
    ├── traceability.json           # TC-ID 追溯映射（供报告生成器使用）
    ├── s1-xxx.yaml                 # 场景1测试文件
    ├── s2-xxx.yaml                 # 场景2测试文件
    └── ...
```

输出路径默认：与功能地图同一目录（用户在步骤 1 指定或从功能地图路径自动推断）

#### 6.2 Markdown 格式

```markdown
# {项目名} - UI 自动化测试用例

生成时间：YYYY-MM-DD
功能地图来源：{功能地图文件名}
功能用例来源：{功能测试用例文件名}
业务系统URL：{base_url}
登录要求：{需要登录：账号 xxx / 不需要登录}

## TC-ID 追溯矩阵

| UI 场景 | 功能 TC-ID | 优先级 | 测试类型 | 覆盖状态 |
|---------|-----------|--------|---------|---------|
| S1: 处罚任务创建 | TC-001, TC-002, TC-003 | P0 | 功能 | ✅ 已覆盖 |
| S2: 任务列表查询 | TC-010, TC-011 | P0 | 功能 | ✅ 已覆盖 |
| — | TC-050 | P1 | API | ❌ 需手工测试 |

## 执行顺序

| 顺序 | 场景 | 依赖 | 页面入口 | 说明 |
|------|------|------|---------|------|
| 1 | S1: ... | 无 | /xxx | ... |

---

## 场景 S1: {场景名称}

**入口URL**: {baseUrl}/xxx
**前置场景**: 无
**为下游提供**: {创建的测试数据描述}

### S1-01: 打开页面并验证列表
- 操作: 导航到 {页面名称}
- 验证: 页面加载完成，列表展示数据

### S1-02: 筛选条件验证
- 操作: 在"{筛选字段名}"下拉框中选择"{选项值}"
- 操作: 点击查询按钮
- 验证: 列表数据根据筛选条件更新

---
```

#### 6.3 suite.yaml 格式

```yaml
name: "{项目名} - UI 测试套件"
base_url: "{base_url}"

login:
  enabled: true/false
  url: "登录页面URL"       # enabled=true 时填写
  steps:                    # enabled=true 时填写
    - action: input
      description: "账号输入框"
      value: "username"
    - action: input
      description: "密码输入框"
      value: "password"
    - action: click
      description: "登录按钮"
    - action: assert
      description: "登录成功，页面跳转"

cases:
  - file: s1-xxx.yaml
    name: "S1: {场景名称}"
    depends_on: []
    entry_url: "/xxx"

  - file: s2-xxx.yaml
    name: "S2: {场景名称}"
    depends_on: ["S1"]
    entry_url: "/xxx"
```

#### 6.4 traceability.json 格式

```json
{
  "project": "{项目名}",
  "generated": "YYYY-MM-DD",
  "source_functional_map": "{功能地图文件路径}",
  "source_test_cases": "{功能测试用例文件路径}",
  "mappings": {
    "S1": {
      "name": "S1: 处罚任务创建",
      "tc_ids": ["TC-001", "TC-002", "TC-003"],
      "priority": "P0",
      "depends_on": [],
      "entry_url": "/penaltyTask/modify-event"
    },
    "S2": {
      "name": "S2: 任务列表查询",
      "tc_ids": ["TC-010", "TC-011"],
      "priority": "P0",
      "depends_on": ["S1"],
      "entry_url": "/penaltyTask"
    }

  "uncovered_tc_ids": [
    {"tc_id": "TC-050", "reason": "API 类型用例，无法 UI 自动化", "priority": "P1"}

#### 6.5 单个场景 YAML 格式

```yaml
name: "S1-{场景名称}"
base_url: "{base_url}"

steps:
  # S1-01: 打开页面并验证列表
  - action: navigate
    url: "/xxx"
    description: "S1-01: 导航到{页面名称}"

  # S1-02: 筛选查询
  - action: select
    description: "{筛选字段名}下拉框选择'{选项值}'"

  - action: click
    description: "点击查询按钮"

  - action: assert
    description: "列表数据根据筛选条件更新"
```

**关键规则**：
1. **不包含登录步骤**：登录由 suite.yaml 统一管理
2. **description 使用功能地图中的实际元素名称**：不是模糊描述
3. **步骤编号格式**：`S{场景号}-{序号}: [步骤名称]`
4. **步骤使用中文描述**：browser-test-agent 使用 AI 执行，自然语言效果更好
5. **URL 使用相对路径**：与 base_url 拼接

## 重要说明

1. **双文件输入**：功能地图提供页面结构和元素信息，功能测试用例提供 TC-ID 和覆盖维度
2. **TC-ID 追溯**：每个 UI 场景必须映射到功能用例 TC-ID，1 个 UI 场景可对应多个功能用例
3. **登录强校验**：步骤 1 必须确认是否需要登录，不可跳过
4. **断言宽泛**：只验证页面状态变化，不写死文案和具体数量
5. **元素名称精确**：操作步骤中使用功能地图中的实际字段名、按钮名、选项名
6. **场景依赖编排**：按功能地图中的业务流程和页面跳转关系排列执行顺序
7. **所有内容使用中文**：场景名称、步骤描述、断言描述
8. **单浏览器会话**：suite.yaml 配置一次登录，所有场景顺序执行
9. **YAML 兼容 p4-browser-test-runner-devtools**：生成的 YAML 可直接用 Chrome DevTools MCP 执行
10. **traceability.json 是报告的关键输入**：p4-browser-test-runner-devtools 读取此文件生成 TC-ID 覆盖报告
