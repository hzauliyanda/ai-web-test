# Web Explorer — 网站功能探索器

通过 Chrome DevTools MCP 控制浏览器，系统性探索 Web 项目并生成功能地图。

## 执行流程

### 第一步：收集信息

用 AskUserQuestion 一次性询问以下内容：

1. **目标 URL** — 要探索的网页地址
2. **是否需要登录** — 是/否
3. **探索深度** — 完整探索（推荐，含子页面、下拉选项、详情页）/ 仅当前页面 / 自定义
4. **输出路径** — 项目测试输出目录（如用户未指定，提示用户选择或创建一个目录），功能地图保存为 `{项目名}功能地图.md`

### 第二步：准备浏览器

1. 调用 `list_pages` 检查 Chrome DevTools MCP 连接
2. 如果连接失败，依次执行：
   - `killall -9 "Google Chrome"` 杀掉所有 Chrome 进程
   - 以调试模式重启：
     ```bash
     /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
       --remote-debugging-port=9222 \
       --user-data-dir=/tmp/chrome-debug \
       "{目标URL}" &
     ```
   - 等待 5 秒后创建 DevToolsActivePort 文件：
     ```bash
     printf "9222\n/devtools/browser/\n" > \
       "$HOME/Library/Application Support/Google/Chrome/DevToolsActivePort"
     ```
   - 再次 `list_pages` 确认连接成功

### 第三步：处理登录

- **需要登录**：`navigate_page` 打开目标 URL → `take_screenshot` 截图展示给用户 → 告知"请在浏览器中手动登录，完成后回复'登录成功'" → 等待用户回复
- **不需要登录**：直接打开目标 URL 继续探索

**绝不替用户输入密码**，因为实际项目可能有 SSO、验证码、2FA。

### 第四步：确认探索范围（强制，不可跳过）

**这一步必须执行，绝不能默认把左侧导航的所有页面全部探索。** 左侧导航页面通常很多、数据量很大，必须先让用户从左侧菜单里挑出要探索的特定页面。

1. **`take_snapshot`** 抓取当前页，**完整枚举左侧菜单栏的所有页面**（包括展开所有可折叠分组、二级/三级子菜单，不要遗漏）
2. 把枚举到的页面**整理成带编号的纯文本清单**列给用户，格式如下（保留菜单层级）：

   ```
   左侧菜单共探测到 N 个页面，请回复要探索哪些（填编号或名称，可多选）：

   1. 一级菜单A
      2. └ 子页面A-1
      3. └ 子页面A-2
   4. 一级菜单B
      5. └ 子页面B-1
   ...
   ```

   > **不要用 AskUserQuestion** 来做这一步 —— 它每题最多只有 4 个选项，装不下完整菜单。必须用纯文本清单让用户自由回复编号/名称。

3. 询问用户：「请回复要探索哪些页面（编号或名称，可多选）；以及这些页面是否需要点进详情页/表单、列表是否要翻页抓全量。」
4. **只探索用户明确指定的页面**，其余一律跳过。用户没有回复指定范围前，**不得开始第五步**。若用户回复"全部"，再确认一次数据量后才继续。

> 即使用户在第一步选了"含详情页/表单"，也必须在本步把左侧菜单清单列出来让用户做减法，只挑特定页面。

### 第五步：探索页面

用 TaskCreate 追踪每个页面的探索进度。**只对第四步用户勾选的页面**执行：

1. **`take_snapshot`** — 获取页面结构（优先，信息更结构化）
2. **`take_screenshot`** — 辅助截图验证
3. **展开所有下拉框** — `click` combobox → 记录 listbox 内所有选项 → `click` 任意选项关闭
4. **进入子页面** — 仅当用户在第四步选择需要深度探索时，点击"查看"/"新增"/"编辑"等按钮，探索详情页和表单页
5. **切换 Tab 页签** — 点击所有 tab 获取不同内容
6. **`navigate_page`** — 仅访问第四步用户勾选的导航页面，**不勾选的页面不访问**
7. **`wait_for`** — 等待页面加载完成再获取内容
8. **主动操作所有功能按钮（含增/改/删）** — 见下方「第五步附：操作型功能探索」，确保不只是"看到"按钮，而是真正"走通"每个功能

### 第五步附：操作型功能探索（增 / 改 / 删 / 提交）

当页面存在"新增 / 创建 / 编辑 / 删除 / 提交 / 发布"等会改变数据的按钮时，**不能只截图记录控件，要实际操作一遍把完整流程走通**，确保所有功能都被探索到。

#### ⛔ 数据安全铁律（最高优先级，任何情况都不得违反）

> **所有"编辑、删除、修改状态、提交、发布"等会改动已有数据的操作，只能在"本次探索中由你自己新增出来的数据"上进行。绝对禁止编辑、删除或修改页面上原有的任何记录、信息、配置。**

具体规则：

1. **先建后改**：要探索"编辑/删除/详情"等行操作，**必须先用"新增/创建"功能造一条自己的测试数据**，然后只在这条自建数据的行上点击编辑/删除/查看。
2. **测试数据可识别**：新增数据时，名称/标题等关键字段统一带显式标记，便于识别和事后清理，例如 `自动探索_勿用_{YYYYMMDD}`、描述写"功能地图探索测试，请勿使用"。
3. **绝不触碰原有记录**：原列表里已存在的记录一律只读——可以看、可以进详情查看结构，但**不点它们的编辑、不点它们的删除、不改它们的任何字段**。
4. **造不出数据就不删改**：如果某页面没有"新增"入口（无法造自己的数据），则该页面的"编辑/删除"功能**只记录按钮存在和位置，不实际执行**，并在功能地图中标注"未实操（无自建数据，避免影响原数据）"。
5. **破坏性操作谨慎**：删除自建数据可以执行（用于探索删除流程）；但任何"批量操作""清空""导出全部"等影响范围大的按钮，先停下来问用户是否执行。

#### 操作流程

1. **新增/创建流程要走完整**：
   - 点击"新增/创建"进入表单或向导，逐字段填入带标记的测试数据。
   - 多步向导（Step1→Step2…）要**逐步走完**：注意很多系统"下一步"需先点"保存"才解锁；跨步时可能弹"有未保存数据，是否继续"确认框——这些真实交互都要记录。
   - 下拉、单选、上传等控件，填值时顺带记录其可选项/类型。
   - 走到最终"提交/完成/保存"，记录成功提示、跳转去向、生成的数据 ID。
2. **编辑流程**：在**刚自建的那条数据**上点"编辑"，记录编辑表单与新增表单的差异（哪些字段变只读、预填了什么）。
3. **删除流程**：在**刚自建的那条数据**上点"删除"，记录二次确认弹窗、删除后列表变化（用于探索完即清理测试数据）。
4. **每一步用 `take_snapshot` 记录真实控件与交互序列**（字段、按钮、弹窗、校验、状态变化），这是后续生成测试用例/自动化脚本的真实依据。

#### 探索后清理

- 探索过程中自建的测试数据，在记录完删除流程时顺便删掉；若未走删除流程，**在第六步结束时主动询问用户是否清理自建的测试数据**，并列出造了哪些数据。

需要记录的信息：

| 维度 | 内容 |
|------|------|
| 页面清单 | 每个页面的名称、URL 路径、页面类型 |
| 导航体系 | 全局导航、子导航、面包屑 |
| 筛选条件 | 字段名、类型、所有下拉选项值 |
| 数据列表 | 列名、操作按钮、按钮状态规则 |
| 分页信息 | 总条数、每页条数 |
| 表单字段 | 字段名、类型、是否必填、默认值 |
| 操作流程 | 新增/编辑/删除等功能的完整操作序列、多步向导各步、保存/提交交互、二次确认弹窗、成功提示与跳转 |
| 页面跳转 | 页面之间的链接和跳转关系 |
| 枚举值 | 所有可能的选项值和状态值 |
| 状态流转 | 数据状态的流转路径 |

### 第六步：生成功能地图

按 [references/output-template.md](references/output-template.md) 中的模板结构生成 Markdown 文件。

写入后执行 `chmod 777 {文件路径}`。

## 关键原则

- **探索前必须让用户框定范围** — 绝不默认抓取左侧导航的全部页面，第四步的范围确认不可跳过
- **⛔ 增删改只在自建数据上操作** — 编辑/删除/提交等改动数据的操作，只能作用于本次探索中自己新增的测试数据，**绝不碰页面原有的任何记录/信息**（详见「第五步附」数据安全铁律）
- **功能要真正走通，不只是看到** — 有新增/编辑等按钮时，要实际操作一遍把完整流程（含多步向导）走完，确保所有功能都被探索到
- 先 snapshot 后 screenshot — snapshot 信息更结构化
- 下拉选项必须展开获取完整列表
- 操作按钮状态规则要记录（哪些状态下禁用）
- 自建测试数据要带显式标记，探索完主动清理或询问用户
- 每步用 TaskCreate 追踪，不丢进度
